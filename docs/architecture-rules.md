# Architecture Rules

## Purpose

This document defines the architectural rules that must be respected across the project.

Its purpose is to ensure:
- consistent layering
- maintainable code organization
- clear dependency direction
- low coupling between components
- safe evolution of the system over time

These rules apply to all manually written code and to any code generated by AI agents.

## Architectural Style

The project follows a pragmatic Clean Architecture approach.

The system is organized into these layers:
- domain
- usecase
- ports
- infra
- delivery

The goal is to keep business logic isolated from frameworks, storage details and external data sources.

## Layer Responsibilities

### Domain
- core business concepts and domain models
- no database code
- no HTTP code
- no infrastructure dependencies

### Use Case
- application logic and workflows
- orchestrates domain objects and ports
- depends only on domain and ports

### Ports
- interfaces used by use cases
- contracts, not implementations

### Infrastructure
- PostgreSQL repositories
- HTTP CSV readers
- technical adapters
- implements ports

### Delivery
- HTTP handlers
- request validation
- response mapping
- must not contain business logic

## Dependency Direction

Allowed:

```text
delivery -> usecase -> ports <- infra
usecase -> domain
ports -> domain
```

Forbidden:
- domain -> infra
- domain -> delivery
- usecase -> infra
- usecase -> delivery
- ports -> infra
- ports -> delivery

## Business Logic Placement Rules

Business logic must live in `usecase` or `domain`.

Never place business logic in:
- HTTP handlers
- repository implementations
- CSV readers
- migration files

## Data Access Rules

All database access must be isolated in repository implementations inside `infra`.

Rules:
- use cases access persistence only through ports
- SQL lives in infrastructure, not in use cases or delivery
- repositories should return domain-friendly results

## External Source Rules

All source-specific ingestion logic must be isolated under `infra/sources`.

Rules:
- source readers parse external formats
- source readers do not persist data directly
- source readers do not orchestrate full ingestion workflow

## Error Handling Rules

- infrastructure returns technical errors with context
- use cases interpret or propagate errors
- delivery maps errors to HTTP status codes
- errors must not be silently ignored
- ingestion failures must be recorded in `ingestion_runs`

## Testing Rules

Critical flows must have tests:
- CSV parsing
- idempotent ingestion
- aggregate query correctness

## Final Rule

If a change does not respect layer boundaries or dependency rules, it must be revised.
